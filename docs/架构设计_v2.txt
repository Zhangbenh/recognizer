## 便携式离线植物识别器（含区域采样系统）

## 1. 文档目标

本文档在 V1 的基础上完成评审修订，目标是形成一份：

- **与需求和系统说明一致**
- **可直接指导实现**
- **可支撑验收**
- **适合个体开发者维护**

的完整架构设计。

本文档重点解决 V1 中存在的以下问题：

1. 采样模式架构闭环不完整
2. 状态机定义不够实现级
3. 地图/区域配置模型缺失
4. 记录模型与统计模型语义不清
5. 错误回退路径不完整
6. 性能目标缺乏可验证机制

## 2. 架构目标与设计原则

### 2.1 架构目标

系统需要在 Raspberry Pi 4B 2GB 上稳定运行，并满足以下目标：

1. 支持离线植物识别
2. 支持普通模式与采样模式
3. 支持地图/区域选择与区域内连续识别
4. 支持本地 JSON 持久化与重启后恢复
5. 支持统计查看
6. 满足单线程、低复杂度、可维护的实现要求
7. 支持对关键验收指标进行验证，包括响应时间、推理时延、稳定性与错误处理

---

### 2.2 设计原则

#### 2.2.1 状态机唯一控制流

所有系统流程必须由状态机驱动。  
任何模块不得绕过状态机直接修改业务流程、页面跳转或持久化行为。  
该原则用于保证：

- 状态互斥
- 可预测的流程控制
- 错误可回退
- UI 与状态一致

#### 2.2.2 单线程主控

主业务逻辑采用单线程同步执行。  
不引入多进程、消息总线、异步任务系统。  
原因：

- 系统规模小
- 交互流程线性
- 设备资源有限
- 单线程更易验证和排错

#### 2.2.3 适配器隔离

所有外部依赖通过适配层抽象，至少包括：

- 相机
- 输入
- 推理
- 存储
- 配置加载

这样做的目的不是“为了优雅”，而是为了控制硬件/框架耦合，避免后续替换 Picamera2、GPIO 或 TFLite 时波及业务层。

#### 2.2.4 UI 完全由状态驱动

UI 不保存独立业务状态。  
UI 只根据当前状态和状态上下文进行渲染。  
不允许出现“状态已切换，但页面仍显示旧内容”的情况。

#### 2.2.5 受控数据增长

系统采用本地 JSON 持久化，但必须显式控制增长边界：

- 单 Region 最大植物种类数 ≤ 50
- 单植物 count ≤ 999
- 单 JSON 文件最大大小 ≤ 1MB

达到上限时必须停止相应写入并提示，而不是静默失败。

#### 2.2.6 Fail Fast + 安全回退

所有关键失败必须：

1. 立即停止当前非法流程
2. 显式提示用户
3. 返回安全状态

不允许失败后进入不确定状态。

---

## 3. 需求与约束基线

本架构直接以以下内容为基线：

### 3.1 功能基线

系统必须支持：

- 普通模式：取景 → 拍照 → 推理 → 显示 → 返回
- 采样模式：首页 → 地图 → 区域 → 识别 → 自动记录 → 循环
- 区域内自动记录
- 本地统计查看
- 本地 JSON 持久化

### 3.2 状态与流程约束

系统说明文档要求：

- 同时仅允许一个状态活跃
- Inferencing / Recording 不允许中断
- Display >5 秒自动返回
- Inferencing >3 秒失败提示
- Recording >1 秒失败提示
- UI 由状态驱动
- 无并发写入

### 3.3 数据基线

当前数据集为 30 类植物，训练集 21000 张、验证集 3000 张，类别均衡，每类训练集 700 张、验证集 100 张。该事实约束了标签映射与模型输出维度设计。

负责：

* 识别结果语义
* 置信度判定

#### Infrastructure Layer

负责：

* GPIO 输入
* 相机采集
* TFLite 推理
* JSON 读写
* 配置文件加载
* 标签文件加载

## 6. 核心模块设计

### 6.1 AppController

#### 职责

* 系统启动入口
* 初始化依赖
* 装配模块
* 启动主循环
* 管理全局生命周期

#### 说明

AppController 不做业务判断，只负责：

* 初始化顺序控制
* 失败时进入启动错误页
* 启动后将控制权交给 StateMachine

---

### 6.2 StateMachine

#### 职责

* 保存当前状态
* 处理事件
* 执行状态迁移
* 调用对应 Use Case
* 保证状态互斥
* 记录状态转移日志

#### 约束

* 同一时刻只允许一个状态活跃
* 所有页面切换必须由状态切换触发
* 非法事件必须被显式忽略或转换为错误提示

---

### 6.3 StateContext

#### 职责

保存状态机运行时上下文，不属于 UI，不直接持久化。

#### 包含内容

```text
mode                      # normal / sampling
selected_map_id
selected_region_id
last_captured_frame
last_recognition_result
last_error
display_deadline
current_stats_snapshot
```

#### 说明

StateContext 是状态机的运行上下文，不是领域存储对象。
它只用于支撑流程推进。

---

### 6.4 InputAdapter

#### 输入事件统一模型

```text
CONFIRM_PRESS
NAV_PRESS
BACK_LONG_PRESS
TIMEOUT
SYSTEM_ERROR
BOOT_OK
BOOT_FAIL
CAPTURE_OK
CAPTURE_FAIL
INFER_OK
INFER_FAIL
RECORD_OK
RECORD_FAIL
```

#### 实现

* GPIOButtonAdapter
* KeyboardAdapter（开发调试用）

#### 说明

用户物理输入只产生原始按键事件，状态机消费的是**归一化事件**。

---

### 6.5 CameraAdapter

#### 接口

```text
start_preview()
capture_frame()
stop_preview()
health_check()
```

#### 实现

* Picamera2CameraAdapter

#### 职责

* 启动预览
* 抓取识别帧
* 健康检查
* 提供统一图像对象给推理模块

#### 关键要求

* 预览流与抓帧流程必须共享同一相机控制链路，避免重复初始化造成延迟
* 拍照后的识别输入必须来源于“当前画面附近帧”，以支持冻结误差 ≤1 帧的验收目标

---

### 6.6 InferenceAdapter

#### 接口

```text
load_model()
infer(frame)
health_check()
```

#### 实现

* TFLiteInferenceAdapter

#### 职责

* 模型加载
* 输入预处理
* 推理执行
* 输出原始 logits/probabilities

---

### 6.7 RecognitionService

#### 职责

* 原始输出解析
* Top-1 结果提取
* 置信度转换
* 阈值判断
* 生成统一 RecognitionResult

#### 判定规则

* 当 confidence < 60% 时，输出“未识别”
* 当 confidence ≥ 60% 时，输出类别结果

#### 说明

该规则来自用户故事中的拒识机制，必须放入领域层，而不是散落在 UI 或存储逻辑中。

---

### 6.8 LabelRepository

#### 职责

* 加载标签映射
* 校验标签总数与模型输出维度一致
* 提供 `class_index -> plant_key/display_name` 映射

#### 约束

由于当前数据集为 30 类植物，标签映射必须保证：

* 类别数量 = 30
* 索引连续
* 与模型输出维度一致

#### 启动失败条件

若标签文件损坏、缺失、数量不匹配，则视为启动失败，进入启动错误页。

---

### 6.9 SamplingConfigRepository

#### 职责

加载地图和区域静态配置。

#### 配置内容

```text
maps:
  - map_id
    display_name
    regions:
      - region_id
        display_name
```

#### 约束

* Map 数量 ≥ 3
* 每个 Map 的 Region 数量 ≥ 4
* 配置为静态只读
* 不支持运行时新增、删除、编辑

#### 启动失败与运行降级

* 若配置文件存在但非法：启动失败
* 若配置文件合法但 maps 为空：系统仍可启动，但采样模式入口显示“无可用地图”

---

### 6.10 StorageAdapter

#### 职责

* JSON 读取
* JSON 保存
* 原子写入
* 文件大小检查
* 数据格式校验

#### 接口

```text
load_region_stats(region_id)
save_region_stats(region_id, data)
```

#### 说明

StorageAdapter 只做基础 IO 与一致性保障，不负责业务规则。

---

### 6.11 SamplingRecorder

#### 职责

* 根据识别结果更新区域统计
* 执行去重与计数
* 处理容量上限
* 生成新的 RegionStatsSnapshot
* 调用 StorageAdapter 持久化

#### 关键业务规则

* 唯一键：`plant_name + region_id`
* 同一植物重复识别不新增条目，`count +1`
* 即使短时间重复识别也计数
* 单次识别最多更新一次记录
* 仅当识别成功且 confidence ≥ 60% 时记录

#### 说明

这里显式采用系统说明文档中的唯一键语义，而不是 V1 中模糊的 `plant_key + region_id`。如果内部实现使用 plant_key，也必须保证其与 plant_name 的持久化语义一致。

---

### 6.12 StatisticsQueryService

#### 职责

* 查询某 Region 的统计结果
* 输出统计页需要的数据视图

#### 输出内容

* 植物种类总数
* 每种植物的数量
* 最近更新时间（可选扩展）

#### 说明

将“统计展示”从存储层中分离出来，避免 UI 直接解析原始 JSON。

---

### 6.13 UIRenderer

#### 页面

* Booting
* Home
* MapSelect
* RegionSelect
* Preview
* InferencingOverlay
* Display
* Stats
* Error

#### 原则

* UI 由状态 + StateContext 驱动
* UI 不持有独立业务状态
* UI 不直接访问 StorageAdapter / InferenceAdapter

## 7. 配置与数据架构

### 7.1 静态配置文件

#### 7.1.1 system_config.json

保存：

* 拒识阈值
* 自动返回时间
* 性能阈值
* 日志级别

#### 7.1.2 labels.json

保存：

* 类别索引
* plant_key
* display_name

#### 7.1.3 sampling_config.json

保存：

* map 列表
* region 列表

---

### 7.2 持久化数据文件

#### 存储目录

```text
data/
  regions/
    region_<region_id>.json
```

#### 每个 Region 一个文件

原因：

* 与“数据按 Region 独立存储”一致
* 简化加载和写入边界
* 降低单文件损坏影响范围

---

### 7.3 持久化模型

#### RegionStatsFile

```json
{
  "region_id": "r1",
  "plant_count": 3,
  "items": [
    {
      "plant_name": "banana",
      "display_name": "Banana",
      "count": 5,
      "last_seen_at": "2026-04-17T14:20:31"
    }
  ],
  "updated_at": "2026-04-17T14:20:31"
}
```

#### 说明

这里明确：

* 持久化的是**区域统计结果**
* 不是完整识别流水日志
* `last_seen_at` 表示该植物最近一次被识别成功的时间
* `updated_at` 表示该 Region 文件最近一次被更新的时间

这样既满足统计需求，也符合“去重但 count+1”的规则。

---

### 7.4 容量与增长控制

#### 约束

* 单 Region 最大植物种类数：50
* 单植物最大 count：999
* 单 JSON 文件最大大小：1MB

#### 行为规则

| 场景           | 行为      |
| ------------ | ------- |
| 新植物超过 50 种   | 拒绝新增并提示 |
| count 已达 999 | 不再增长    |
| 文件即将超过 1MB   | 停止写入并提示 |

#### 说明

这是领域层规则，由 SamplingRecorder 判定，由 StorageAdapter 协助执行。

## 8. 状态机设计

### 8.1 状态列表

正式状态统一定义为：

* `BOOTING`
* `HOME`
* `MAP_SELECT`
* `REGION_SELECT`
* `PREVIEW`
* `CAPTURED`
* `INFERENCING`
* `DISPLAY`
* `RECORDING`
* `STATS`
* `ERROR`

#### 说明

相比 V1：

* 补足了启动态与错误态
* 统一使用名词/形容词式状态名
* 消除 `Capture` / `Captured`、`Record` / `Recording` 混用问题

---

### 8.2 状态进入动作

| 状态            | 进入动作                       |
| ------------- | -------------------------- |
| BOOTING       | 初始化配置、标签、相机、模型、存储          |
| HOME          | 渲染模式选择，默认高亮普通模式            |
| MAP_SELECT    | 加载 map 列表并高亮第一项            |
| REGION_SELECT | 加载所选 map 的 region 列表并高亮第一项 |
| PREVIEW       | 启动或保持相机预览                  |
| CAPTURED      | 固化当前识别帧到 StateContext      |
| INFERENCING   | 调用推理服务并启动超时计时              |
| DISPLAY       | 显示识别结果并启动 5 秒自动返回计时        |
| RECORDING     | 在采样模式下写入统计并启动 1 秒超时计时      |
| STATS         | 加载并显示当前 Region 统计结果        |
| ERROR         | 显示错误信息并等待可恢复输入             |

---

### 8.3 状态退出动作

| 状态          | 退出动作                   |
| ----------- | ---------------------- |
| PREVIEW     | 无，保持相机链路               |
| CAPTURED    | 无                      |
| INFERENCING | 清理本次推理临时对象             |
| DISPLAY     | 清理展示定时器                |
| RECORDING   | 刷新 StateContext 中的统计快照 |
| ERROR       | 清理当前错误信息               |

---

| ------------- | --------------- | ----------------- | ---------- |

### 8.5 非法事件处理规则

| 状态          | 非法事件处理      |
| ----------- | ----------- |
| INFERENCING | 所有用户输入忽略    |
| RECORDING   | 所有用户输入忽略    |
| DISPLAY     | 除返回类输入外其余忽略 |
| BOOTING     | 用户输入忽略      |
| ERROR       | 仅允许返回安全页或重试 |

---

### 8.6 状态超时规则

| 状态          | 超时阈值 | 行为             |
| ----------- | ---- | -------------- |
| INFERENCING | >3 秒 | 触发 INFER_FAIL  |
| RECORDING   | >1 秒 | 触发 RECORD_FAIL |
| DISPLAY     | >5 秒 | 触发 TIMEOUT     |

## 9. 核心业务流程

### 9.1 普通模式识别流程

```text
HOME
→ PREVIEW
→ CAPTURED
→ INFERENCING
→ DISPLAY
→ PREVIEW
```

#### 详细说明

1. 用户在首页选择普通模式
2. 进入 PREVIEW，显示实时取景与准星
3. 按确认键后抓取当前帧
4. 进入推理
5. 生成识别结果
6. 在 DISPLAY 中显示植物名称/置信度或“未识别”
7. 5 秒后自动返回 PREVIEW，或长按返回

该流程覆盖用户故事中的普通模式闭环。

---

### 9.2 采样模式识别流程

```text
HOME
→ MAP_SELECT
→ REGION_SELECT
→ PREVIEW
→ CAPTURED
→ INFERENCING
→ DISPLAY
→ RECORDING
→ PREVIEW
```

#### 关键规则

* 只有在采样模式下才允许 RECORDING
* 只有在识别成功且置信度达到阈值时才记录
* 若结果为“未识别”，DISPLAY 结束后直接回 PREVIEW，不进入 RECORDING

这使得“自动记录”规则与“低置信度不记录”规则在架构层闭环。

---

### 9.3 统计查看流程

```text
REGION_SELECT / PREVIEW
→ STATS
→ PREVIEW 或 REGION_SELECT
```

#### 展示内容

* 当前 Region 的植物种类数
* 每个植物的数量
* 无数据时显示空态

这与用户故事中的数据查看要求一致。

## 10. 识别链路设计

### 10.1 识别处理管线

```text
Preview Frame
→ Capture Frame
→ Preprocess
→ TFLite Inference
→ Postprocess
→ Label Mapping
→ Threshold Decision
→ RecognitionResult
```

### 10.2 RecognitionResult 定义

```text
plant_name
display_name
confidence
is_recognized
```

#### 说明

* `plant_name`：用于持久化与规则判断
* `display_name`：用于 UI 展示
* `confidence`：标准化后的置信度百分比
* `is_recognized`：领域层明确标记，避免 UI 再次判断

---

### 10.3 标签映射规则

由于当前数据集是 30 类平衡分类任务，LabelRepository 必须保证模型输出索引与标签文件一一对应。若输出维度与标签数不一致，则模型部署无效。

---

### 10.4 置信度规则

* Top-1 类别作为候选输出
* 置信度低于 60% 时统一视为“未识别”
* UI 不参与阈值判断
* SamplingRecorder 只接受 `is_recognized = true` 的结果

该方案可满足先前确定的两键 + 长按返回基线，并兼顾设备简化。

用户故事中的“单按键完成核心操作”应解释为：

## 13. 错误处理与恢复设计

### 13.1 错误分类

* `CameraError`
* `InferenceError`
* `StorageError`
* `DataError`
* `ConfigError`
* `ModelError`
* `LabelError`

---

### 13.2 错误处理策略

统一原则：

```text
Fail Fast
→ Show Error
→ Return Safe State
```

---

### 13.3 具体错误回退路径

| 错误             | 触发时机       | 回退策略                  |
| -------------- | ---------- | --------------------- |
| CameraError    | 启动或抓帧      | 进入 ERROR；不可进入 PREVIEW |
| ModelError     | 模型加载失败     | 启动失败，进入 ERROR         |
| LabelError     | 标签缺失/数量不符  | 启动失败，进入 ERROR         |
| ConfigError    | 采样配置非法     | 启动失败，进入 ERROR         |
| DataError      | 读取 JSON 失败 | 显示错误；统计不可用；识别功能可继续    |
| StorageError   | 写入失败       | 显示错误；不阻塞后续返回 PREVIEW  |
| InferenceError | 推理异常/超时    | 显示错误；返回 PREVIEW       |

这里对“存储失败”和“推理失败”的处理与系统说明一致：
不允许卡死，不允许留在不确定状态。

---

### 13.4 安全状态定义

本系统安全状态包括：

* `HOME`
* `PREVIEW`
* `ERROR`

原因：

* HOME：用户可重新选择流程
* PREVIEW：普通识别流程的稳定待机状态
* ERROR：显式承载不可恢复故障信息

## 14. 性能与可验收设计

### 14.1 性能预算

用户故事和系统说明给出了关键时延目标：拍照 ≤0.2s、推理 ≤2.5s、全流程 ≤3s。

本架构将其拆解为：

| 阶段         | 预算     |
| ---------- | ------ |
| 按键到抓帧开始    | ≤100ms |
| 抓帧完成       | ≤100ms |
| 推理         | ≤2.5s  |
| 后处理 + 显示准备 | ≤200ms |
| 总计         | ≤2.9s  |

这样可为轻微系统波动预留 100ms 缓冲。

---

### 14.2 启动预算

用户故事要求系统初始化后 2 秒内进入取景界面。

因此建议启动路径分为：

#### 必要初始化

* system_config
* labels
* model
* camera health check
* input init

#### 可延后初始化

* 各 region 数据懒加载
* stats 页面数据按需加载

这样可以避免启动阶段被无关 IO 拖慢。

---

### 14.3 可观测性设计

为支撑验收与调试，系统必须记录最少量运行日志：

#### 启动日志

* 模型加载耗时
* 标签加载耗时
* 相机初始化耗时
* 配置加载结果

#### 流程日志

* 状态迁移
* 抓帧耗时
* 推理耗时
* 写入耗时
* 错误事件

#### 建议输出格式

```text
[timestamp] [state] [event] [duration_ms] [result]
```

#### 价值

这不是“锦上添花”，而是为了证明：

* 开机是否在 2 秒内进入取景
* 推理是否 ≤2.5 秒
* Recording 是否 ≤1 秒
* 连续 30 次操作是否稳定

## 17. 风险清单与缓解

### 17.1 关键风险

#### 1. 置信度映射风险

模型若输出为量化 uint8 或未经 softmax 处理的分值，可能导致“60% 阈值”解释错误。
**缓解**：在 InferenceAdapter 中固定后处理策略，并通过离线样本验证。

#### 2. 标签映射风险

模型输出索引与标签文件若不一致，会直接导致识别类别错误。
**缓解**：启动时校验标签数量与输出维度。当前应为 30 类。

#### 3. 相机与 UI 兼容风险

树莓派显示链路与相机预览可能出现刷新、拉伸或性能问题。
**缓解**：优先验证 Preview 页帧率与抓帧延迟，先打通最小预览闭环。

#### 4. JSON 断电一致性风险

写文件时异常中断可能造成损坏。
**缓解**：采用临时文件 + rename 原子写入。

#### 5. GPIO 长按误触发风险

按键抖动、长按时长判定不稳可能造成返回误触发。
**缓解**：统一去抖与长按识别时序。

#### 6. 树莓派 2GB 内存风险

模型、预览帧缓存、UI 资源叠加后可能造成内存波动。
**缓解**：控制图像中间对象生命周期，避免重复缓存全尺寸帧。

#### 7. 启动时间超标风险

若统计数据在启动时全量加载，可能影响 2 秒内进入取景。
**缓解**：Stats 数据按需加载。

## 18. 与验收标准的映射

| 验收要求           | 架构保障机制                             |
| -------------- | ---------------------------------- |
| 开机后 2 秒内进入取景   | 启动路径精简、延迟加载 stats                  |
| 取景帧率 ≥15 FPS   | Preview 独立渲染预算、相机预览链路常驻            |

| 全流程 ≤3s        | 全链路预算拆分                            |
| 置信度 <60% 输出未识别 | RecognitionService 统一阈值规则          |
| 低置信度不记录        | SamplingRecorder 仅接收 recognized 结果 |

## 19. 最终架构结论

本系统的最终架构定义为：

> 一个基于**单线程状态机驱动**的离线植物识别与区域采样系统；
> 其以**适配器隔离外部依赖**，以**状态驱动 UI**，以**领域规则约束识别与记录**，以**本地 JSON 实现受控持久化**，并通过**显式错误回退与最小可观测性设计**支撑实现与验收。
