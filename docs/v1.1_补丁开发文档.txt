# 《v1.1 补丁开发文档》
## Recognizer v1.1 Patch Development Document

---

## 0. 文档信息

- 文档版本：v1.1-final
- 文档日期：2026-04-23
- 适用对象：产品设计、架构设计、状态机设计、实现开发、测试验收、部署交付
- 文档定位：用于正式归档 v1.1 补丁的产品边界、架构补丁、状态机补丁、实现计划与验证方案

---

## 1. 文档目的

本文件用于把现有 Recognizer 系统从“离线植物识别器”重定义为“云端优先、本地回退的植物识别与区域采样系统”，并将 v1.1 版本所需的新增能力、架构改造、状态机改造、配置调整、资源要求、开发计划与验收口径一次性固定下来。

本文件的目标是：

- 明确 v1.1 的产品边界与功能范围
- 给出可直接落地的代码改造方向
- 统一后续文档修订口径
- 为开发、测试、部署提供单一参考基线

---

## 2. 版本定位

### 2.1 v1.0 与 v1.1 的关系

v1.0 已完成并交付，核心能力为：

- 普通模式植物识别
- 采样模式地图/区域选择与区域统计
- 本地模型推理
- 本地 JSON 持久化

v1.1 不是简单缺陷修复，而是一次带产品边界调整的补丁版本。该版本在保留现有单线程状态机框架和本地数据结构的基础上，新增如下能力：

- 默认中文显示
- 地图页与区域页图片化选择
- 百度植物识别 API 云端优先识别
- 本地模型自动回退
- 地图级汇总统计页

### 2.2 v1.1 的产品重定义

自 v1.1 起，系统定义更新为：

> 一个基于单线程状态机驱动、支持中文界面、支持云端优先识别并在失败时回退本地模型、支持区域采样与地图级汇总统计的植物识别系统。

### 2.3 v1.1 对旧版“离线-only”定义的处理

以下旧约束在 v1.1 中不再作为产品定义硬边界：

- 仅允许本地模型识别
- 系统目标仅定义为离线运行
- 性能预算仅围绕本地推理口径定义

但以下实现原则继续保留：

- 单线程事件驱动运行时
- 状态机为唯一控制流入口
- JSON 作为采样统计持久化介质
- 物理输入仍为双键语义

## 3. v1.1 目标与范围

### 3.1 版本目标

v1.1 的主要目标为：

1. 把用户可见界面统一切换为中文
2. 优化地图页和区域页的可识别性与可操作性
3. 提升识别成功率，减少本地模型单独工作时的误识别与未识别
4. 保持 v1.0 已交付主流程的稳定性与数据兼容性
5. 在不推翻现有状态机框架的前提下完成可维护扩展

### 3.2 本版本包含的能力

- 普通模式：云端优先识别，本地回退
- 采样模式：云端优先识别，本地回退，自动记录
- 中文默认文案
- 中文植物、地图、区域展示名
- 地图页图片化选择
- 区域页图片化选择
- 地图级植物聚合统计页
- 保留区域级统计页

### 3.3 本版本明确不做的内容

- 不新增触摸操作
- 不新增第三个物理按键
- 不将本地 JSON 替换为数据库
- 不支持运行时动态新增地图或区域
- 不新增跨所有地图的全局总统计页
- 不在 v1.1 中强制执行“历史云端统计键自动归并到本地标签目录”的迁移任务

## 4. 关键决策冻结清单

以下为 v1.1 已确认决策：

### 4.1 中文显示策略

- 默认 UI 语言为中文
- v1.1 只要求“显示名称本地化”
- 本地标签目录中已有的 `plant_key` 与 `plant_name` 保持现有内部语义
- 对于本地标签目录中不存在的云端植物，允许生成稳定的云端合成 `plant_key`
- 对于本地标签目录中不存在的云端植物，`display_name` 与 `plant_name` 可直接使用百度返回的中文名
- `display_name` 改为中文用于 UI 展示
- 本版本不要求提供完整的运行期中英切换界面

### 4.2 图片资源策略

- 地图和区域缩略图由项目内静态资源提供
- 缩略图路径写入配置文件
- screen 后端负责图片展示
- text 后端保留文本列表，作为测试与联调用兜底输出

### 4.3 云端识别策略

- 百度植物识别 API 在普通模式与采样模式均启用
- 识别策略为“云端优先，本地回退”
- 云端网络失败、鉴权失败、超时、返回异常时立即回退本地
- 云端返回名称若可映射到本地标签目录，则优先使用本地 `plant_key`
- 云端返回名称若无法映射到本地标签目录，则生成云端合成 `plant_key` 后允许直接写入统计
- 仅当云端返回空名称、非法名称或合成键生成失败时才回退本地
- 本地回退后仍走现有统一识别结果结构

### 4.4 地图汇总统计策略

- 新增独立状态 `MAP_STATS`
- 入口定义为 `MAP_SELECT + NAV_LONG_PRESS -> MAP_STATS`
- 页面内容为“当前地图下所有区域按植物聚合后的统计”
- 不采用“按区域列摘要”作为 v1.1 主方案

## 5. 产品行为定义（v1.1）

### 5.1 普通模式

普通模式主流程保持为：

```text
HOME -> PREVIEW -> CAPTURED -> INFERENCING -> DISPLAY -> PREVIEW
```

其中识别策略变为：

```text
capture frame
-> try cloud recognition
-> if cloud success and name mapped: use cloud result
-> else fallback local model
-> build unified RecognitionResult
```

### 5.2 采样模式

采样模式主流程保持为：

```text
HOME -> MAP_SELECT -> REGION_SELECT -> PREVIEW
-> CAPTURED -> INFERENCING -> DISPLAY -> RECORDING -> PREVIEW
```

识别来源虽改为“云端优先，本地回退”，但 `RECORDING` 的入库规则不变：

- 只有识别成功的结果可记录
- 记录以统一统计键 `plant_key` 为唯一业务键
- 若云端结果已映射到本地标签目录，则沿用本地 `plant_key`
- 若云端结果未映射到本地标签目录，则使用稳定的云端合成 `plant_key`
- 统计仍按 region_id 聚合存储

### 5.3 地图汇总统计

新增地图级统计主流程：

```text
MAP_SELECT -> MAP_STATS -> MAP_SELECT
```

其目标是提供“当前地图所有区域的植物聚合视图”，便于用户在地图层级查看总体采样情况。

## 6. UI 与交互补丁设计

### 6.1 中文默认显示

以下界面需改为中文：

- 启动页
- 主菜单页
- 地图选择页
- 区域选择页
- 预览页提示文案
- 识别结果页
- 区域统计页
- 地图统计页
- 错误页

### 6.2 本地化边界

为降低改动风险，v1.1 只本地化以下内容：

- 植物 `display_name`
- 地图 `display_name`
- 区域 `display_name`
- 所有硬编码提示文案

以下内容不变：

- 本地标签目录中既有的 `plant_key`
- 本地标签目录中既有的 `plant_name`
- `map_id`
- `region_id`
- 现有 JSON 存储键结构继续以 `plant_key` 为记录键，但允许新增云端命名空间键

### 6.3 地图页图片化选择

地图页的目标交互为：

- 页面顶部显示当前页面标题与地图名称
- 中间展示当前高亮地图的大图或主卡片
- 底部或侧边展示 1 行缩略图
- 高亮项有边框或反色提示
- `NAV_PRESS`：切换到下一张地图
- `CONFIRM_PRESS`：确认当前地图并进入区域页
- `BACK_LONG_PRESS`：返回 HOME
- `NAV_LONG_PRESS`：进入地图汇总统计页

### 6.4 区域页图片化选择

区域页的目标交互为：

- 页面顶部显示当前地图名称
- 中间展示当前高亮区域图片
- 底部或侧边展示该地图下区域缩略图列表
- 高亮项清晰可见
- `NAV_PRESS`：切换到下一张区域图
- `CONFIRM_PRESS`：确认区域并进入 PREVIEW
- `BACK_LONG_PRESS`：返回 MAP_SELECT
- `NAV_LONG_PRESS`：进入区域统计页

### 6.5 地图统计页显示内容

地图统计页建议展示：

- 当前地图中文名称
- 地图下区域总数
- 已产生记录的区域数
- 植物种类总数
- 植物聚合列表（分页）

每条聚合项建议展示：

- 植物中文名
- 该植物在整张地图中的总记录次数
- 覆盖区域数
- 最近一次或最高一次置信度（实现时二选一，需统一）

## 7. 识别架构补丁设计

### 7.1 总体原则

v1.1 不新增“云端推理独立状态”，而是在现有 `INFERENCING` 状态内部完成云端优先与本地回退的编排，理由如下：

- 保持 UI 流程稳定
- 保持状态机复杂度可控
- 降低对现有 handler 与 transition 的侵入
- 保持 text/screen 后端表现一致

### 7.2 推荐识别编排

`RecognitionService` 重构后应承担三类职责：

1. 本地识别路径
2. 云端识别路径
3. 云端失败时的回退编排

推荐逻辑：

```text
recognize(frame)
  -> try recognize_via_cloud(frame)
  -> if cloud result valid and normalized to stats key: return unified result
  -> else recognize_via_local(frame)
```

### 7.3 新增基础组件

推荐新增以下组件：

- `CloudConfigRepository`
- `BaiduTokenRepository` 或等价 token 缓存组件
- `BaiduPlantClient`
- `BaiduPlantMappingRepository`

### 7.4 结果统一要求

无论结果来自云端还是本地，都必须统一生成 `RecognitionResult`，确保：

- 后续 DISPLAY 不关心识别来源
- RECORDING 不关心识别来源
- 统计层只接收统一结构
- 测试可复用现有断言方式
- 对未进入本地标签目录的云端植物也能生成稳定的统计键

### 7.5 识别来源字段

推荐对 `RecognitionResult` 做受控扩展，新增但不限于：

- `source`: `cloud` | `local`
- `fallback_used`: bool
- `raw_label_name`: str | None
- `catalog_mapped`: bool

此字段在 v1.1 中不要求必须显示在 UI，但建议保留用于日志、测试与问题定位。

## 8. 百度云端识别适配策略

### 8.1 接入目标

百度 API 在 v1.1 中承担“先尝试识别”的职责，不直接替代本地模型，但允许在本地标签目录之外扩展统计键体系，以便把百度返回的新增植物中文名直接纳入统计。

### 8.2 接入要求

- 使用 access_token 调用百度植物识别接口
- 运行时完成 token 获取与缓存
- 输入为拍照后的 RGB 帧编码结果
- 调用失败不阻塞系统长时间停留在错误路径

### 8.3 名称归一化与统计入库要求

百度返回的是植物中文名和分数，不一定存在于本系统本地标签目录。为同时满足“优先复用本地标签体系”和“允许云端新增植物直接写入统计”这两个目标，v1.1 采用双层名称归一化策略。

第一层：显式映射到本地标签目录

- 若 `baidu_plant_mapping.json` 中存在“百度中文名 -> 本地 `plant_key`”映射，则优先采用本地 `plant_key`
- 此时 `display_name` 仍按 v1.1 中文显示策略输出，统计会自然并入本地既有植物记录

第二层：生成云端合成 `plant_key`

- 若百度中文名未命中本地映射表，则不再将其视为错误
- 系统应为该中文名生成稳定的云端合成 `plant_key`
- 建议命名空间格式为：`cloud_baidu::<stable_token>`
- `stable_token` 建议由“百度中文名规范化结果 + 可选短哈希”生成，以避免重名与非法字符问题
- 此时 `display_name` 直接使用百度返回的中文名
- 此时 `plant_name` 也可直接使用百度返回的中文名
- 此时 `catalog_mapped=false`

由此，百度返回的植物中文名即使在本地不存在对应 `plant_key`，也可以稳定写入区域统计和地图汇总统计。

归一化要求：

- 有显式本地映射时，优先采用本地 `plant_key`
- 无显式本地映射时，允许直接生成云端合成 `plant_key` 入库
- 仅当百度返回空名称、非法名称、名称规范化失败或合成键生成失败时，才回退本地模型
- 统计层统一按 `plant_key` 字符串聚合，不区分其来自本地目录还是云端扩展目录
- 后续若需要将某个云端合成键归并到本地标签目录，可通过离线迁移脚本处理，但不作为 v1.1 强制范围

### 8.4 错误分类建议

建议新增如下错误类型：

- `CloudNetworkError`
- `CloudAuthError`
- `CloudAPIError`
- `CloudMappingError`

建议处理方式：

- `CloudNetworkError`：记录告警后立即本地回退
- `CloudAuthError`：记录错误后立即本地回退，并在日志中标记配置问题
- `CloudAPIError`：记录错误后立即本地回退
- `CloudMappingError`：仅用于“百度名称为空、非法、规范化失败或合成键生成失败”这类真正无法入库的场景；记录告警后立即本地回退

### 8.5 云端调用超时策略

云端请求必须有严格超时，且其预算必须从整体 INFERENCING 预算内切分，而不是额外叠加。

建议初始预算：

- 云端请求超时：3.0s
- 本地模型推理预算：1.5s
- `INFERENCING` 总体预算：4.5s

上述预算为 v1.1 初始建议值，最终可在真机联调后微调。

## 9. 配置、资源与数据结构补丁

### 9.1 `system_config.json` 新增项

建议新增：

- `ui_language`: 默认 `zh-CN`
- `recognition_strategy`: 默认 `cloud_first`
- `cloud_request_timeout_s`: 默认 `3.0`
- `local_infer_timeout_s`: 默认 `1.5`
- `infer_timeout_s` 调整为整体 `INFERENCING` 预算，默认 `4.5`
- `performance_budget.cloud_success_s`
- `performance_budget.cloud_fallback_s`

### 9.2 新增 `cloud_config.json`

用于承载：

- 百度 API Key
- Secret Key
- token 缓存参数
- 重试参数
- 云端超时参数

要求：

- 支持环境变量覆盖
- 不要求把敏感信息直接提交进版本库

### 9.3 新增 `baidu_plant_mapping.json`

用途：

- 维护百度返回名称到本地 plant_key 的显式映射
- 为后续把云端扩展植物归并进本地标签目录预留配置入口
- 未命中映射表时，不视为非法结果，而是转入云端合成 `plant_key` 生成流程

### 9.4 `labels.json` 调整

调整原则：

- `display_name` 改为中文
- `index` 保持不变
- `plant_key` 保持不变
- `plant_name` 保持原内部语义

### 9.5 `sampling_config.json` 调整

地图与区域对象新增：

- `thumbnail_path`

并将 `display_name` 改为中文。

### 9.6 静态资源目录建议

建议新增：

```text
assets/
  fonts/
  maps/
  regions/
```

其中：

- `fonts/` 存放中文字体
- `maps/` 存放地图缩略图与主图
- `regions/` 存放区域缩略图与主图

## 10. 统计模型补丁设计

### 10.1 区域统计保持不变

v1.0 已有区域统计页继续保留，其职责不变：

- 展示单个区域内的植物统计
- 基于 `region_id` 查询
- 分页显示

### 10.2 地图统计新增能力

地图统计新增的核心能力为：

- 读取整张地图下所有区域记录
- 聚合同一统计 `plant_key` 的所有计数
- 统计该植物覆盖的区域数
- 构建地图级分页快照

### 10.3 数据源设计

地图统计不新增底层持久化文件，直接基于现有：

- `sampling_records.json`
- `RegionStatsRepository.load_all()`

进行内存态聚合。

### 10.4 统计项建议

地图聚合统计项建议最少包含：

- `plant_key`
- `display_name`
- `total_count`
- `covered_region_count`
- `last_confidence`
- `catalog_mapped`

### 10.5 排序策略调整

由于中文 `display_name.lower()` 不再适合作为稳定排序依据，v1.1 建议将统计页排序改为：

- 先展示已映射到本地标签目录的植物，并按 `labels.json` 中的 `index` 排序
- 对未映射到本地标签目录的云端植物，再按 `display_name` 排序

这样可避免中文排序歧义，并保证区域统计页与地图统计页顺序一致。

## 11. 状态机补丁设计

### 11.1 状态集合

v1.1 状态集合建议为：

```text
BOOTING
HOME
MAP_SELECT
MAP_STATS
REGION_SELECT
PREVIEW
CAPTURED
INFERENCING
DISPLAY
RECORDING
STATS
ERROR
```

其中新增状态仅为：

- `MAP_STATS`

### 11.2 新增状态职责

`MAP_STATS` 用于展示当前地图的植物聚合统计，其职责为：

- 读取当前 `selected_map_id`
- 聚合该地图所有区域统计
- 生成分页 view model
- 响应翻页和返回行为

### 11.3 关键转移补丁

建议新增或保留如下规则：

```text
MAP_SELECT + NAV_PRESS -> MAP_SELECT       (切换地图)
MAP_SELECT + CONFIRM_PRESS -> REGION_SELECT
MAP_SELECT + NAV_LONG_PRESS -> MAP_STATS
MAP_SELECT + BACK_LONG_PRESS -> HOME

MAP_STATS + NAV_PRESS -> MAP_STATS         (翻页)
MAP_STATS + BACK_LONG_PRESS -> MAP_SELECT

REGION_SELECT + NAV_PRESS -> REGION_SELECT (切换区域)
REGION_SELECT + CONFIRM_PRESS -> PREVIEW
REGION_SELECT + NAV_LONG_PRESS -> STATS
REGION_SELECT + BACK_LONG_PRESS -> MAP_SELECT

STATS + NAV_PRESS -> STATS                 (翻页)
STATS + BACK_LONG_PRESS -> REGION_SELECT
```

### 11.4 不新增的状态

v1.1 明确不新增如下状态：

- `CLOUD_INFERENCING`
- `LOCAL_FALLBACK`
- `MAP_SUMMARY_LOADING`

原因：

- 会提高状态机复杂度
- 会增加额外 UI 分支
- 当前需求可在现有 `INFERENCING` 中完成

### 11.5 保留的运行时约束

虽然产品定义发生变化，但以下状态机约束继续保留：

- 所有输入事件进入 FIFO 队列
- 禁止绕过状态机直接切换界面
- `INFERENCING` 期间仍不处理额外用户中断
- `RECORDING` 期间仍不处理额外用户中断

## 16. 风险与应对

### 风险 1：中文字体在树莓派不可用

应对：

- 将字体资源纳入项目静态目录
- 启动期校验字体文件是否存在

### 风险 2：百度识别延迟过高

应对：

- 设置严格超时
- 超时后立即回退本地
- 单独记录云端成功路径与回退路径耗时

### 风险 3：百度名称无法覆盖本地 30 类

应对：

- 维护显式映射表
- 未映射时生成稳定的云端合成 `plant_key`
- 仅在名称为空、非法或合成键生成失败时回退本地
- 补充映射覆盖率测试

### 风险 4：图片资源过大导致屏幕渲染卡顿

应对：

- 启动期预加载缩略图
- 控制图片分辨率
- 增加简单缓存策略

### 风险 5：地图统计字段与区域统计字段差异过大

应对：

- 明确地图统计为独立状态和独立 view model
- 共享分页结构，但不强行复用所有字段

## 18. 结论

v1.1 的核心不是“给 v1.0 增加几处零散功能”，而是在保留现有单线程状态机与本地数据结构优势的前提下，完成一次产品能力升级。

该升级的落地原则为：

- 产品层：默认中文、支持云端优先、支持地图级汇总统计
- 架构层：云端优先但本地可回退
- 状态机层：只新增一个必要状态 `MAP_STATS`
- 数据层：保留原有 `plant_key` 主键结构，并允许以云端命名空间键扩展统计目录
- UI 层：screen 后端增强，text 后端稳定
- 测试层：以“回退可验证、数据不破坏、流程不回归”为验收核心

当以上内容完成后，Recognizer v1.1 将具备更高的实际可用性、更好的交互可读性，以及更合理的统计能力扩展空间。
